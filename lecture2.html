Lecture 2: SDLC & General Principles

1. Software Development Life Cycle (SDLC)

Why SDLC?

Without a structured process, team members may start tasks in different orders — one writing code first, another doing design, another testing — leading to project failure. SDLC provides a disciplined and systematic framework for developing software, especially in team environments.

💡 Analogy Building a house requires a fixed order: foundation → walls → roof → electrical → plumbing → finishing. You cannot put the roof before the walls. SDLC gives the same structured order to software development.

The 7 Phases of SDLC

Phase 1: Planning & Requirement Analysis

The most important stage. Business analyst meets client to gather requirements — what to build, who are the end users, project objective. A Feasibility Study assesses whether the project is practical and likely to succeed.

Phase 2: Defining Requirements

The SRS (Software Requirement Specification) document is created. It contains all product requirements to be constructed and developed during the project lifecycle.

Phase 3: Designing the Software

All knowledge from requirements and analysis is used to design the software architecture. Includes database design, UI design, and system architecture.

Phase 4: Developing the Software

Actual coding begins. Developers follow coding guidelines defined by management and use programming tools to build the software.

Phase 5: Testing

Software is tested against requirements to verify it solves the needs gathered during the requirements stage. Bugs are identified and fixed.

Phase 6: Deployment

Once certified bug-free, the software is deployed to production for real users.

Phase 7: Maintenance

Clients start using the system. Real issues emerge and requirements evolve. The team fixes bugs, improves performance, and adds new features over time.

💡 Full SDLC Example — Building an E-Commerce Website
  1. Planning: Client wants an online clothing store. Budget is $50,000. Feasibility check: viable.
  2. Requirements: SRS created: product listing, cart, payment gateway, order tracking.
  3. Design: Database schema, UI wireframes, and API architecture designed.
  4. Development: React frontend, Node.js backend, MySQL database — coded.
  5. Testing: Unit tests, integration tests, load testing with 1000 users.
  6. Deployment: Hosted on AWS, SSL certificate configured, domain connected.
  7. Maintenance: Black Friday bug fixed, new category "Accessories" added.

2. The 7 General Principles of Software Engineering (David Hooker)

David Hooker proposed seven principles that focus on the philosophy behind good software engineering.

Principle 1: The Reason It All Exists

A software system exists for one reason: to provide value to its users. Always ask: "Does this add real value to the system?"

Example: Adding a flashy animation that slows down page load adds no value — remove it.

Principle 2: KISS — Keep It Simple, Stupid!

All design should be as simple as possible, but no simpler. Simple systems are easier to understand, test, and maintain.

Example: A login form needing only email + password is simpler than one requiring 5 fields.

Principle 3: Maintain the Vision

A clear vision is essential to a successful project. Compromising the architectural vision will eventually break even well-designed systems.

Principle 4: What You Produce, Others Will Consume

Always specify, design, and implement knowing someone else will maintain your code. Write with clarity and documentation.

Example: Writing clean, commented code so the next developer can understand it without asking you.

Principle 5: Be Open to the Future

Systems must be ready to adapt to any changes — specifications can change at any moment, and hardware platforms can become obsolete in months.

Principle 6: Plan Ahead for Reuse

Build software so that code and design can be reused. Planning for reuse reduces cost and increases the value of components.

Example: A reusable authentication library can be used across 10 different projects.

Principle 7: Think!

Placing clear, complete thought before action almost always produces better results. If you think and still do something wrong, it becomes a valuable learning experience.
