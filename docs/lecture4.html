Lecture 4: Software Process Models — Waterfall

1. Introduction

A Software Process is a set of related activities that leads to the production of a software product.

A Software Process Model is a simplified representation of a software process — a blueprint for how teams organize their work.

💡 Analogy: A software process model is like a recipe. Just as a recipe shows steps to bake a cake (mix, bake, frost), a process model shows steps to build software.

2. The Waterfall Model

The first and most classic SDLC model. It follows a strictly linear, sequential approach where:

  • The output of the current phase = input for the next phase
  • The next phase starts only after the previous phase is completed and signed off
  • There is no going back in the classic version
💡 Analogy: Like a waterfall — water flows down from one level to the next and cannot flow back up. Once you leave a phase, you cannot return.

Waterfall Phases in Detail

Phase 1: Feasibility Study

Assessment of the practicality of a proposed project. Helps decision-makers determine whether a project or investment is likely to be successful.

Phase 2: Requirements Analysis & Specification

System's services, constraints, and goals are established via consultation with system users. These are defined in detail and serve as the system specification.

Phase 3: System & Software Design

Transform requirements from SRS into a form that permits coding. Define the overall software architecture with high-level and detailed design.

Phase 4: Coding & Unit Testing

System is developed in small programs called units, which are integrated in the next phase. Unit testing is done to verify each unit works correctly.

Phase 5: Integration & System Testing

All units are integrated into a system. The entire system is then tested for any faults and failures across the complete integrated product.

Phase 6: Maintenance & Operation

The longest phase. System is installed and put into practical use. Errors found after release are corrected, system units are improved, and new requirements are addressed.

💡 Waterfall Example — Library Management System
  1. Feasibility: Library has 5,000 books and needs digital cataloging. Budget OK.
  2. Requirements: SRS written: member registration, book issue/return, fine calculation, search.
  3. Design: ER diagram, system flowchart, UI mockups designed.
  4. Coding: Each module coded: MemberModule, BookModule, FineModule.
  5. Testing: All modules integrated & tested together. 12 bugs found and fixed.
  6. Maintenance: After 6 months, an e-book feature is added.

3. When to Use Waterfall Model

  • Requirements are clear, well-documented, and fixed
  • Requirements are unlikely to change radically during development
  • Dynamic/unstable technologies are NOT involved
  • No ambiguous requirements
  • Project is short

4. Advantages of Waterfall Model

  • Simple and easy to understand and use
  • Requirements are simple and explicitly declared
  • Requirements remain unchanged throughout development
  • Start and end points for each phase are fixed — easy to track progress
  • Release date and final cost can be determined before development begins
  • Works well for small projects

5. Disadvantages of Waterfall Model

  • Not suitable for significant and complex projects
  • Cannot accept changes in requirements during development
  • Difficult to go back to a previous phase
  • Testing is done late — risks and challenges aren't identified early
  • No working software produced until late in the lifecycle

6. Iterative Waterfall Model

An improved version of the Classic Waterfall Model. The key difference:

Provides feedback paths from every phase to its preceding phases. When errors are detected at a later phase, the feedback paths allow correcting errors committed during an earlier phase, and these changes are reflected going forward.

Note: There is no feedback path back to the Feasibility Study stage, because once a project has been accepted, it is not easily abandoned.

Advantages:

  • Feedback path allows error correction between phases
  • Simple to understand and use
  • Clear separation of phases for better planning
  • Less time spent on documentation

Disadvantages:

  • All requirements must still be stated before development — hard to incorporate late changes
  • No incremental delivery — customer waits for the full product
  • Overlapping of phases not supported
  • No built-in mechanism for risk handling
  • Limited customer interactions (only at start and end)
