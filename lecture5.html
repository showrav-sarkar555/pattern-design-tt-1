Lecture 5: V-Shaped Model & Incremental Model

1. Verification vs Validation

VerificationValidation
Process of determining if software is designed and developed as per specifications Process of checking if the software (end product) has met the client's true needs and expectations
Are we building the product right? Are we building the right product?
Static analysis method (review) — without executing code Dynamic analysis method — done by executing code
Examples: Code reviews, inspections, walkthroughs Examples: Unit testing, integration testing, system testing, acceptance testing
💡 Example: A hospital software is verified when the design matches the specification documents. It is validated when an actual doctor tests it and confirms it meets their workflow needs.

2. V-Shaped Model (Verification & Validation Model)

The V-Shaped Model is an extension of the Waterfall Model where testing is associated with every phase of the development lifecycle.

Key Characteristics:

  • Linear like Waterfall — next phase starts only after previous phase is complete
  • Testing is planned in parallel with the corresponding development phase
  • Left side (descending): Development phases (Verification)
  • Right side (ascending): Testing phases (Validation)
  • The bottom of the V: Implementation/Coding

The V-Shape Phases

Requirements Analysis ↔ Acceptance Testing

Management understands requirements from the customer's perspective. Acceptance Test Plan is prepared here. After the software is built, the user tests for acceptance. Output: mutually agreed requirements document.

System Design ↔ System Testing

Overall system is designed — high-level picture including user, software, hardware, interfaces, and database. System Test Plan is created at this stage. System testing done just before product release.

Example: Design of the whole 3-story building.

Architecture Design ↔ Integration Testing

High-level software design. All technical decisions made here: programming language, communication protocols, memory. Integration Test Plan created here.

Example: Design of each floor of the building.

Module Design ↔ Unit Testing

Developers design individual modules. Unit Test Plan created here. Low-level design documents generated. Unit tests verify each module works independently.

Implementation (Coding)

The bottom of the V. The actual code is written according to all design documents created in previous phases. This is the only phase where coding happens.

💡 V-Model Example — Online Exam System
  • Requirements: Students answer MCQs, auto-graded. → Plan: user acceptance test
  • System Design: Web-based system, cloud database. → Plan: load test for 500 users
  • Architecture: Python Flask, PostgreSQL, Redis. → Plan: API integration tests
  • Module Design: QuestionModule, TimerModule, GradeModule. → Plan: unit tests per module
  • Coding: All modules coded
  • ← Unit tests executed → Integration tests → System tests → Acceptance tests

When to Use V-Shaped Model

  • Requirements are well-defined and not ambiguous
  • Small to medium-sized projects with clearly defined and fixed requirements
  • When sample technical resources with essential technical expertise are available

Advantages:

  • Simple and easy to understand
  • Testing methods (planning, test design) happen well before coding — saves time
  • Higher chance of success compared to Waterfall
  • Avoids downward flow of defects
  • Works well for small plans where requirements are easily understood

Disadvantages:

  • Not suitable for complex and object-oriented projects
  • If requirements change mid-way, test documents AND requirement documents must both be updated
  • Requires crystal-clear documentation
  • Very rigid and least flexible

3. Incremental Model

In the Incremental Model, requirements are divided into multiple standalone modules. The software is built and delivered in increments — each increment adds functionality to the previous one.

How It Works:

  • Requirements divided into multiple standalone modules
  • Module-by-module working approach
  • Every subsequent release adds function to the previous release
  • Each module goes through: communication → planning → modeling → construction → deployment
  • First increment is often a core product (basic functionality)
  • Customer feedback from core system is addressed during later increments
  • Process repeated after delivery of each increment until complete product is produced
  • Maximum customer interaction throughout
💡 Incremental Model Example — Food Delivery App
  • Increment 1 (Core): Restaurant listing + order placement — delivered & tested
  • Increment 2: Payment gateway integration — delivered & tested
  • Increment 3: Real-time order tracking with GPS — delivered & tested
  • Increment 4: Reviews and ratings system — delivered & tested
  • Increment 5: Loyalty points & promo codes — delivered & tested
Each increment is a working product. Customers use it and give feedback for the next increment.

When to Use Incremental Model

  • Project has a lengthy development schedule
  • Software team is not very well skilled or trained
  • Customer demands a quick release of the product
  • When you want to deliver prioritized requirements first

Advantages:

  • Errors are easy to recognize and fix
  • Easier to test and debug smaller increments
  • Tolerates changing requirements
  • Balances cost and manpower
  • Risk is managed during each iteration
  • Client gets important functionality early — no need to wait for full product

Disadvantages:

  • Needs good planning to divide requirements into increments
  • Actual cost may exceed the estimated cost
  • A problem in one unit needs to be corrected in all units — time consuming
